# Mythical Man-Month

## Chapter 2

소프트웨어 프로젝트가 부족한 시간때문에 망가지는 이유;

1. 소요시간 추정 능력 부족
2. 많은 인원을 투입한다고 빠르게 끝나는 것이 아니다.
3. 관리자도 확신이 없기에 철저하게 잡지 않는다.
4. 일정 진척이 제대로 모니터링되지 않는다.
5. 시간 부족을 해결하기 위한 인력 추가 투입은 사태를 오히려 악화시킨다.

### 낙관주의

**"이번엔 정말 제대로 돌아갈 겁니다."**

- 모든 작업이 예정된 시간 내 완료될 것이라는 가정은 잘못되었다.
- 아이디어를 구현하는 데는 시간과 노력이 든다.
- 컴퓨터 프로그래밍의 재료가 유연하다는 것이 구현이 쉽다는 것을 의미하지 않는다.
- 우리의 아이디어 자체에 흠이 있다.

=> 우리의 낙관주의에는 정당성이 없다.

### ManMonth

man-month; 투입된 노력을 셀 때 쓰는 단위, 투입된 사람 수 * 달 수

-> 하지만, 사람과 일정은 교환 가능하지 않다!
유일한 경우 : 서로 간 소통할 필요가 없고, 완전히 분할가능한 경우

하지만, 소프트웨어 개발 작업은 분할이 어렵다.

=> 커뮤니케이션으로 추가되는 부담 : 훈련, 의사소통\
사람을 더 투입하는 것은 일정을 더 늘어지게 만든다.

### 시스템 테스트

계획 수립 : 1/3\
코딩 : 1/6\
구성 요소 테스트와 초기 시스템 테스트: 1/4\
모든 구성 요소가 준비 된 후 시스템 테스트: 1/4

=> 계획 수립에 많은 시간 배정, 전체 일정 절반에 디버깅 할당, 코딩은 추정하기 쉬운 부분이므로 적게 할당

결국 시스템 테스트를 일정의 절반에 사용하게 된다. 이 때의 일정 지연은 심리적, 재정적 영향이 막대하므로 2차적 비용이 더 크다. 초기 일정 수립시에 테스트에 충분한 시간을 할애해야 한다.

소심한 추정(버그 발생률 등)치를 마련하고, 공유해야 한다.

### 일정 붕괴의 악순환

1개월로 가정했던 첫번째 마일스톤이 2개월째 지속되고 있다고 가정하자. 이 때 관리자가 취할 수 있는 대안들은?

1. 작업을 그대로 두면서 원래 계획대로 일정에 맞추기 위해 추가 인력 투입
2. 일정 재수립
3. 작업 범위 축소

=> "늦어진 소프트웨어 프로젝트에 인력을 추가로 투입하면 더 늦어지게 된다."
